# OMP 发布可靠性补丁设计

**日期：** 2026-07-11
**状态：** 已实现

## 背景

发布验收发现三个相互独立但都影响发布可信度的问题：项目范围资源操作没有携带工作区标识；发布报告记录的是 Pi 依赖版本而非实际 OMP 运行时版本；本地化增量检查在未提供比较基线的干净 CI 工作区中只检查最后一个提交。

本补丁将三项修复合并交付，但不改变 OMP 的资源配置格式、既有 IPC 名称或安装产物结构。

## 目标与边界

目标：

- 资源中心对项目范围 MCP、Skills 和 Agents 的创建、编辑、启停、删除、测试均在当前工作区执行。
- Release QA 报告准确标识已校验并打包的 OMP 二进制版本。
- CI 对整个待合并变更集执行本地化硬编码检查，不允许静默降级为仅检查最后一个提交。

非目标：

- 不迁移或重写既有用户、项目资源配置。
- 不改变 OMP 二进制的打包来源或版本选择策略。
- 不将本地化检查扩展为全仓历史代码清理。

## 设计

### 资源操作上下文

`OmpResourceDirectory` 已拥有可选的 `workspaceId`，刷新请求也已使用它。所有变更型资源请求统一追加同一字段：创建、更新、启停、删除，以及 MCP 连接测试。该字段保持可选，以支持无工作区的用户范围操作；选择 `project` 或操作项目条目时，后端据此解析项目根目录与配置文件。

现有后端 handler、DTO 和服务层继续使用 `input.workspaceId`，不新增 RPC channel。

### 运行时版本证据

质量门禁中的运行时能力检查负责解析实际打包路径，并对已解析的 OMP 二进制执行 `omp --version`。该检查应把该命令成功输出的版本作为结构化结果返回；`release:win` 报告直接消费该值。禁止从 `package.json` 中的 Pi 依赖推断 OMP 版本，因为两者独立发布。

如果版本探测失败，运行时能力检查失败，发布命令保持 fail-fast，不生成成功报告。

### CI 本地化比较基线

检查脚本保留开发者模式：不设置比较基线时，优先检查工作区相对于 `HEAD` 的改动。

在 CI 模式中，脚本必须取得明确的合并基线：优先使用 `I18N_BASE_REF`，其次使用 CI 提供的目标分支信息并计算 merge-base。若均不可用则失败并给出配置提示，不能退化为 `HEAD~1`。比较命令采用 `base...HEAD`，保证覆盖 PR 中所有提交。调用 `validate:ci` 的环境需提供可解析的目标分支引用。

## 验证

- 为资源目录组件补充断言：每一种资源操作都携带 active workspace ID。
- 为质量命令补充测试或可注入命令断言，确认报告中的版本来自 OMP 二进制探测结果。
- 为本地化变更检查补充多提交/merge-base 覆盖，验证 CI 无基线时失败、提供基线时覆盖完整范围。
- 运行相关单元测试、`bun run quality:quick`、`bun run lint:i18n:changed`，并执行 TypeScript typecheck。
